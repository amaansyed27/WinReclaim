# Release Engineering

Official WinReclaim releases are built on GitHub-hosted Windows runners and published through the **Release Windows** workflow in `.github/workflows/release-windows.yml`.

Do not publish official installers from an arbitrary local machine or manually replace release assets after `latest.json` has been generated.

## Release outputs

A stable Windows release should contain:

```text
WinReclaim_*_x64-setup.exe
WinReclaim_*_x64-setup.exe.sig
WinReclaim_*_x64*.msi
WinReclaim_*_x64*.msi.sig
latest.json
```

Exact filenames are generated by Tauri and can vary with bundle configuration.

The recommended consumer download is the NSIS setup executable. MSI is available for managed or administrative deployment.

## Updater endpoint

Installed applications read:

```text
https://github.com/amaansyed27/WinReclaim/releases/latest/download/latest.json
```

The manifest references the signed Windows installer. Tauri verifies the signature using the public key embedded in `src-tauri/tauri.conf.json`.

## Versioning

WinReclaim uses semantic versions:

```text
MAJOR.MINOR.PATCH
```

Examples:

```text
1.2.0
1.2.1
1.3.0-beta.1
```

Enter the version in the workflow without the `v` prefix. The workflow creates the `vX.Y.Z` tag.

Do not manually create the tag before running the workflow. The workflow refuses to reuse an existing tag.

## One-time updater signing setup

Tauri updater signatures are mandatory for official updates.

Repository Actions secrets:

- `TAURI_SIGNING_PRIVATE_KEY` — complete contents of the updater private key;
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` — set only when the key is encrypted.

The matching public key is committed in `src-tauri/tauri.conf.json`.

### Key custody rules

- Never commit the private key.
- Never paste it into an issue, log or pull request.
- Never attach it to a GitHub Release.
- Keep at least one protected offline backup.
- Restrict repository secret administration.
- Do not regenerate the key after users install a release unless a signed migration strategy exists.

Existing installations trust the embedded public key. Losing the private key prevents future trusted updates. Leaking it allows an attacker to sign malicious updater artifacts.

Tauri updater signing is not the same as Windows Authenticode code signing. An installer can pass Tauri update verification while Windows still displays an unknown-publisher or SmartScreen warning.

## Pre-release checklist

Before starting the workflow:

1. Confirm `main` contains the intended code and documentation.
2. Confirm CI is green.
3. Confirm no other release workflow is running.
4. Confirm the new semantic version does not already exist as a tag.
5. Review `CHANGELOG.md` and prepare concise release notes.
6. Verify the updater secret is present.
7. Confirm `src-tauri/tauri.conf.json` contains the public key matching the secret.
8. Run manual safety scenarios from [testing.md](testing.md).
9. Verify optional model/runtime provenance when those files changed.
10. Avoid merging unrelated changes while the release workflow is between checkout and its version commit.

The final point matters because the workflow can update version manifests and push a commit to `main`. A concurrent commit can cause a non-fast-forward failure.

## Running the release

1. Open the repository **Actions** tab.
2. Select **Release Windows**.
3. Confirm the filename shown by GitHub is `release-windows.yml`.
4. Select **Run workflow** on `main`.
5. Enter the new version without `v`.
6. Add release notes.
7. Enable prerelease only for test builds.
8. Run the workflow.

The workflow performs these stages:

1. checks out `main` with full tag history;
2. validates semantic version syntax and tag availability;
3. verifies the updater signing secret;
4. installs locked frontend dependencies;
5. synchronizes npm, Cargo and Tauri versions;
6. validates frontend integrity and production build;
7. runs Rust formatting, tests and strict Clippy;
8. commits synchronized version manifests when necessary;
9. builds NSIS and MSI bundles;
10. signs updater artifacts;
11. creates the version tag and GitHub Release;
12. uploads installers, signatures and `latest.json`.

## Post-release verification

Do not treat a green workflow alone as sufficient. Verify the public release.

### Release page

Confirm:

- the release is not a draft;
- prerelease state is correct;
- tag and title match the intended version;
- both installer formats are present;
- corresponding signatures are present;
- `latest.json` is present;
- asset sizes are plausible;
- release notes describe consequences and requirements accurately.

### Installer smoke test

On a clean or disposable Windows test environment:

1. download the NSIS setup from the release page;
2. install as a standard user;
3. launch WinReclaim;
4. run a Quick scan;
5. verify findings and protected items;
6. execute a narrow reversible test action;
7. inspect the receipt;
8. restore from the vault;
9. uninstall and check expected local-data retention.

Repeat MSI installation when MSI distribution changed.

### Updater test

The complete updater test requires two versions:

1. install the previous stable release;
2. publish the new stable release;
3. open the previous version;
4. run update check;
5. verify version, notes and download;
6. install the update;
7. relaunch and confirm the new version;
8. confirm local snapshots, receipts, vault and optional models behave according to migration policy.

Installing the same version does not validate the updater path.

### Manifest verification

Retrieve the public manifest:

```powershell
Invoke-RestMethod `
  https://github.com/amaansyed27/WinReclaim/releases/latest/download/latest.json |
  ConvertTo-Json -Depth 10
```

Confirm:

- `version` matches the release;
- the Windows platform entry exists;
- the URL points to the uploaded official asset;
- the signature is non-empty;
- the asset can be downloaded;
- no private or temporary URL is present.

## Release failure handling

### Tag already exists

Choose a new version. Do not force-move a published release tag.

### Version commit cannot push

Synchronize `main`, ensure no concurrent release is running and rerun with the same version only if the tag/release was not created. Inspect workflow steps before retrying.

### Build succeeds but publication fails

Do not manually assemble `latest.json` without understanding the signing relationship. Inspect bundle files and action logs, fix the workflow and rerun safely.

### Signature missing or invalid

Stop the release. Verify the private secret and embedded public key match. Never publish an unsigned update or bypass updater verification.

### Partial release exists

If the tag/release exists with incomplete assets:

- determine whether any user could have downloaded it;
- avoid silently replacing artifacts referenced by an existing manifest;
- prefer a new patch version when integrity or provenance is uncertain;
- document the incident in release notes when needed.

## Prereleases

Use prerelease builds for limited testing. A prerelease should:

- use a semantic prerelease suffix when appropriate;
- be marked as a GitHub prerelease;
- avoid becoming the stable `/releases/latest` target;
- use clear test-only notes;
- still use valid signatures and normal safety checks.

## Forks

A fork must use:

- its own application identifier where appropriate;
- its own release endpoint;
- its own updater key pair;
- its own protected secret storage;
- clear unofficial branding.

Reusing WinReclaim's public key without the matching private key produces unusable updates. The official private key is never shared.

## Release documentation

For each release:

- move relevant `CHANGELOG.md` entries from **Unreleased**;
- update version-specific documentation only after publication;
- keep source/docs claims aligned with the shipped installer;
- record security fixes without exposing users prematurely;
- update model/runtime notices when provenance changes.
